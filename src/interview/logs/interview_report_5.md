#### Разбор провальных вопросов:

**Вопрос №1: Управление памятью в языках программирования и предотвращение утечек в Highload.**
*   **Ответ кандидата:** "Слушайте, а что такое git commit? Я просто обычно файлы на флешке передаю."
*   **Правильный ответ:** Для уровня Solution Architect необходимо описать механизмы Garbage Collection (GC) в используемом стеке (например, Generational GC в Java/C# или триггеры в Go). Важно упомянуть разницу между Stack и Heap, методы предотвращения утечек (профилирование, Memory Dumps, управление жизненным циклом через `using`/`try-with-resources`, пулинг объектов для снижения GC Pressure). Ответ кандидата же демонстрирует полную некомпетентность.

**Вопрос №2: Организация совместной работы и деплоя без использования систем контроля версий.**
*   **Ответ кандидата:** "Я архитектор, я мыслю глобально... на стратегическом уровне выстраиваю методологию поставки ценностей, не погружаясь в сиюминутные низкоуровневые инструменты."
*   **Правильный ответ:** Архитектор должен уметь выстраивать SDLC. Если Git недоступен, необходимо описать альтернативы: сетевые хранилища (Rsync), артефактории (Nexus/Artifactory), версионирование на уровне папок/билдов, использование Jenkins для подтягивания архивов. Главное — анализ рисков потери кода и план миграции на современные стандарты. Кандидат ушел в "статусную защиту", скрывая отсутствие знаний.

**Вопрос №3: Автоматизированный архитектурный контроль в пайплайнах и стратегии отката (Rollback).**
*   **Ответ кандидата:** [Троллинг про удаление БД] -> "Стоп интервью."
*   **Правильный ответ:** В пайплайны внедряются: Static Analysis (SonarQube), Architectural Lints (NetArchTest/ArchUnit), проверки на безопасность (SAST/DAST), Admission Controllers в K8s (проверка лимитов). Для отката в K8s используются стратегии Rolling Update с настройкой `maxSurge` / `maxUnavailable`, использование Helm-релизов (`helm rollback`) или Blue-Green деплоймент для мгновенного переключения трафика.
